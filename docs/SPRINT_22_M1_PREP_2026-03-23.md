# Sprint 22 M1 Split Prep (No Rollout)

## Objective

Prepare modular split for workloads without changing active production runtime path.

Target workloads:
- `tga-ingest`
- `tga-stage5`
- `tga-stage6`
- `tga-web`
- `tga-mcp`

## Ownership Mapping

## tga-ingest
- Telegram session/connectivity ownership
- listener/bot/backfill/archive ingestion workers
- produces messages/events for downstream processing

## tga-stage5
- Stage 5 analysis/extraction pipeline
- embeddings/merge/metrics workers coupled to Stage 5 data flow
- must not run Telegram session ownership logic

## tga-stage6
- strategy/draft/review/domain orchestration execution flows
- no Telegram connectivity ownership
- isolated from ingestion uptime risks

## tga-web
- web read/review/ops/search surfaces
- no ingestion and no Stage 5 ownership by default

## tga-mcp
- standalone MCP TypeScript service
- isolated container/network behavior maintained

## Startup Role Mapping Prep

Prepared role tokens for host startup selection:
- `ingest`
- `stage5`
- `stage6`
- `web`
- `ops`
- `maintenance`
- `mcp` (reserved marker for split coordination)
- `all` (default for backward compatibility)

Notes:
- current default remains `all`, preserving existing behavior
- role isolation is prep-only until explicit rollout window after Stage 5 tail

## Compose/Runtime Prep (Docs-Only)

Planned service split is documented as operational intent only:
- define per-workload startup commands and ownership in runbook form
- stage compose/service split in a dedicated rollout window after Stage 5 tail
- keep current `docker-compose.yml` path unchanged until rollout gate is opened

## Rollout Guard

Do not activate split until:
1. current Stage 5 tail completes
2. Sprint 20 guards are shipped and stable
3. Sprint 21 role-aware wiring is validated in staging-like environment
4. ownership conflict checks are green (no double-running workers)

## Post-tail Verification

- each workload starts with intended role set
- only ingest owns Telegram session
- Stage 5 starts without stage6/web/mcp coupling
- health checks pass per workload
- no duplicate ownership across services
