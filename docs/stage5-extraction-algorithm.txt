Stage5 extraction algorithm (as-is snapshot)
Timestamp (UTC): 2026-03-17

Current runtime shape
- Session-first analysis is canonical path.
- AnalysisWorkerService processes cooled chat sessions from `chat_sessions` queue.
- Cheap extraction is chunked per session and per model; expensive resolver handles backlog with retries/backoff.
- DialogSummaryWorkerService works on slices (chat_sessions), not on message-id watermarks.

------------------------------------------------------------
1) Main loop order (AnalysisWorkerService)
------------------------------------------------------------
Each iteration executes in this order:
1. Process expensive backlog first.
2. Process messages marked needs_reanalysis.
3. Process session-first queue:
   - `chat_sessions` where `is_analyzed=false`, `is_finalized=false`,
     `last_message_at <= now - SessionAnalysisMinIdleMinutes`.
4. Seed/refresh session boundaries from processed backlog.

For each analyzable session:
- load messages in session boundaries
- split session into adaptive chunks (`BuildAdaptiveSessionChunks`)
- process cheap extraction for each chunk
- on successful completion: ensure fast per-slice summary, mark session analyzed

Progress state:
- `stage5:session_watermark_ms`
- `stage5:session_seed_message_watermark`
- `stage5:session_chunk_checkpoint:{chatId}:{sessionIndex}`

------------------------------------------------------------
2) Input eligibility for cheap pass
------------------------------------------------------------
Session-level eligibility:
- session is cooled down by `SessionAnalysisMinIdleMinutes`
- session is not finalized
- session is not analyzed yet

Message-level skip guard inside cheap pass:
- empty semantic content
- service placeholders (`[DELETED]`, `[SERVICE_*]`)
- technical noise (service/system chatter)

Episodic guard:
- blocks messages for slices whose previous slice has no summary
- can quarantine repeatedly skipped messages by policy

------------------------------------------------------------
3) Context building for cheap/extractive calls
------------------------------------------------------------
Per message context includes:
- reply context (`reply_to_message_id`)
- local burst context
- session start context
- historical summary context
- optional previous-slice summary prompt for session continuity

Payload is assembled by `MessageContentBuilder.BuildCheapBatchPrompt(...)`.

------------------------------------------------------------
4) Cheap model routing and LLM calls
------------------------------------------------------------
Routing:
- non-AB: all messages -> `Analysis.CheapModel`
- AB: deterministic bucket by message id (`message_id % 100`) between baseline/candidate

Chunking:
- split by model
- adaptive chunking by char budget and hard limits
- one OpenRouter call per cheap chunk (`phase=cheap`)

Failure behavior:
- chunk failures logged to `extraction_errors` (`stage5_cheap_batch_model`)
- optional one-by-one fallback for transient/model failures
- failed messages marked for reanalysis

------------------------------------------------------------
5) Post-processing and persistence
------------------------------------------------------------
For each processed message:
1. normalize/sanitize/refine extraction
2. validate schema/content constraints
3. upsert `message_extractions.cheap_json`
4. set `needs_expensive` when required
5. if expensive not required -> apply extraction transactionally to domain tables

Domain writes are handled by `ExtractionApplier`.

------------------------------------------------------------
6) Expensive pass resolver
------------------------------------------------------------
Backlog source:
- `message_extractions` where `needs_expensive=true`
- retry gate (`expensive_next_retry_at`)

Behavior:
- budget check (`ExpensiveDailyBudgetUsd`)
- provider/model fallback/backoff
- finalize with cheap extraction when retries exhausted or denied

------------------------------------------------------------
7) Slice summary worker (DialogSummaryWorkerService)
------------------------------------------------------------
Summary worker now consumes slices from `chat_sessions`:
- source: `GetPendingAggregationCandidatesAsync(staleBeforeUtc)`
- hot/cold gate: 15-minute idle timeout (`HotSessionIdleTimeout`)
- per-slice checkpoint key:
  `stage5:summary:session:{chatId}:{sessionIndex}`

For each eligible slice:
- load slice messages by session boundaries
- filter technical noise/trash-only sessions
- generate summary with `SummarizeDialogAsync` (scope `episodic_slice`)
- fallback to deterministic summary when message count is too low or LLM fails
- upsert `chat_sessions.summary`
- upsert session summary embedding for retrieval

Important:
- Legacy message-id summary watermarks (`stage5:summary_watermark`,
  `stage5:summary_extraction_watermark`) are not part of active runtime flow.

------------------------------------------------------------
8) Cold path and finalization
------------------------------------------------------------
`DailyKnowledgeCrystallizationWorkerService` aggregates cooled session summaries
into final daily summaries and marks archived sessions/final states.

------------------------------------------------------------
9) Operational checks
------------------------------------------------------------
- Session queue pressure:
  `chat_sessions where is_analyzed=false and is_finalized=false`
- Cheap usage:
  `analysis_usage_events where phase='cheap'`
- Summary usage:
  `analysis_usage_events where phase='summary'`
- Expensive backlog:
  `message_extractions where needs_expensive=true`
